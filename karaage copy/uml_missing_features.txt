# UML図 精密精査報告書 - 機能不足と拡張提案

現在のUpStreamアプリケーションにおけるUMLモデリング機能を、プロフェッショナルな設計ツールとしての観点から再精査しました。

---

## 1. 共通基盤機能（すべてのUML図に適用すべき事項）
- [ ] **ノードのネスト（包含関係）のセマンティクス**:
    - 現在、システム境界やパッケージは単なる「見た目の枠」ですが、枠内の図形を枠と一緒に移動したり、枠のプロパティとして内部要素を認識する機能が必要です。
- [ ] **コネクションの直交ルーティングの最適化**:
    - 図形を回避する自動経路計算（障害物回避）の実装。
- [ ] **ノート（Note）のアンカー機能**:
    - ノートから対象の図形へ、点線で自動追従するアンカー線の実装。
- [ ] **多重度・役割名の端点表示**:
    - 線の端付近に「0..1」「*」「+owner」などのテキストを個別に配置・保持する機能。

---

## 2. ユースケース図 (Use Case Diagram)
- [ ] **アクターのバリエーション**:
    - 外部システム（四角形に «system»）、時間アクター（時計アイコン）などの視覚的区別。
- [ ] **ユースケースの拡張点 (Extension Points)**:
    - 楕円の中に横線を引き、その下に拡張条件（condition）や拡張箇所をリスト表示する機能。
- [ ] **包含・拡張のセマンティクス**:
    - 矢印の中央に自動で «include», «extend» を配置し、かつ拡張(extend)は矢印の向きが逆（拡張元から基底へ）になる等のルール強制。

---

## 3. クラス図 (Class Diagram)
- [ ] **詳細なメンバー表現**:
    - 静的(static)メンバーの「下線」表示。
    - 抽象(abstract)メソッドの「斜体」表示。
- [ ] **インターフェース表記の切り替え**:
    - 通常のクラスボックス形式と、簡略化された「ロリポップ（○―）」および「ソケット（⊂―）」形式の相互変換。
- [ ] **高度な関係性**:
    - 限定子 (Qualifier): 関連の端に付く小さな四角形。
    - 関連クラス (Association Class): 線から点線でクラスボックスが伸びる形式。
- [ ] **テンプレートクラス**:
    - クラスボックスの右上に点線の四角で型パラメータ（<T>等）を表示。

---

## 4. アクティビティ図 (Activity Diagram)
- [ ] **制御フローとデータフローの区別**:
    - 矢印の先端の形状による区別。
- [ ] **ピン (Pins)**:
    - アクションの境界に配置され、オブジェクトが流れるポート。
- [ ] **パラメータノード**:
    - アクティビティ全体の枠の境界に配置される入出力ポート。
- [ ] **割込可能な領域 (Interruptible Region)**:
    - ギザギザの点線枠で囲まれた領域。

---

## 5. ステートマシン図 (State Machine Diagram)
- [ ] **状態の内部記述 (Internal Transitions)**:
    - entry / exit / do / on event などの動作を状態枠内に整理して表示。
- [ ] **疑似状態の網羅**:
    - ジャンクション (Junction): 実行時に判定される分岐。
    - 終了状態 (Terminate): インスタンスが消滅する。
    - 入場点・退場点 (Entry/Exit Point): 複合状態の境界に付く小さな白丸/黒丸。

---

## 6. シーケンス図 (Sequence Diagram) ※新規実装案
- [ ] **ライフラインの動的生成**:
    - クラス図で定義したクラスをライフラインとして配置。
- [ ] **同期・非同期メッセージの区別**:
    - 塗りつぶし矢印（同期）と、開いた矢印（非同期）の切り替え。
- [ ] **実行仕様（実行バー）の自動計算**:
    - メッセージの送受信タイミングに合わせてライフライン上のバーを自動伸縮。
- [ ] **複合フラグメント (alt, loop, opt, ref)**:
    - 領域を区切り、ガード条件を書けるようにする。

---

## 7. 実装優先度の提案
1. **中核**: シーケンス図の新規実装（上流設計において最も需要が高いため）。
2. **改善**: クラス図のメンバー編集機能の高度化（静的/抽象などの修飾子対応）。
3. **洗練**: 全図面共通の「図形回避ルーティング」と「ネスト移動」の改善。
